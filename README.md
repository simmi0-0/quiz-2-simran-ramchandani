part b:
- navigation pattern: the expected result is that User can access
different dashboard sections easily. the pre- requisite is that User is logged in the redirected to dashboard. the test case is that User clicks the “schedule” or “settings” in the sidebar. action result is that User is lead to corresponding dashboard section
- feedback and status
    1. expected result: User sees current and past service status clearly
    2. Pre-requisite: User has completed or scheduled
    3. test case: Dashboard displays “next appointment” and recent “recent history” panels
    4. action result: User sees service type, data and status (e.g. completed, upcoming)
- input and forms:
  1. expected result: User can initiate a new service booking
  2. pre-requisite: User is on dashboard and wants to book a service
  3. test case: User clicks “+ book a new service panel”
  4. action result: User is redirected to booking form with input fields
- layout and structure:
   1. expected result: User can scan dashboard quickly due to panel-based layout
   2. pre-requisite: User has service history and upcoming appointment
   3. test case: User input needed data to fill the form
   4. action result: User can visually separate tasks and navigate without confusion
- data representation:
    1. expected result: User sees service data organized in readable panels
    2. pre requisite: User has service history and upcoming appointments
    3. test case: Dashboard shows service type, date, status, and action button in each panel
    4. action result: User can interpret service info and click “manage” or “details” for more actions
